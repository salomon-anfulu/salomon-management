## 二、问题清单（按优先级排列）

### P0 — 高优先级（建议近期处理）

#### 1. 数据量持续增长，18个月后撑满 localStorage

**现状**：当前 localStorage 数据量 100.5KB，月增长约 285KB
- linggongAttendance: 每月新增 ~274 条 × ~500字节 ≈ 137KB
- attendance（同步生成）: ~80KB/月
- doorSchedule + storeSupport: ~45KB/月

**风险**：约18个月后达到 5MB localStorage 上限，用户数据会被截断或报错

**建议**：
- **短期**（现在）：实现月度数据归档——把3个月前的灵工打卡和考勤数据从运行时移出到独立的 `data/archive/` 目录，页面改为按月加载
- **中期**（3个月内）：考虑迁移到 IndexedDB（上限远大于5MB），或引入轻量后端API

#### 2. 版本检查条件堆积（23条技术债）

**现状**：`Store.init()` 中累积了 23 条 `isOutdated*` 条件，每次更新数据就加一条，这些条件永远不会被删除（因为不确定用户是否已刷新过）

```js
// 当前：一条IF判断里塞了23个 || 条件
if (isOldFormat || isOutdatedAvail || isOutdatedAvail2 || isOutdatedDoor || 
    hasBrokenTime || isOutdatedLG || isOutdatedRatings || isOutdatedPerf || 
    isOldStaffFormat || isOutdatedDengRating || isOutdatedJunePerf || 
    isOutdatedKXY || isOutdatedRatingsV2 || isOutdatedDoorV2 || isOutdatedRatingsV4 || 
    isOutdatedRatingsV5 || isOutdatedRatingsV6 || isMissingReviews || 
    isOutdatedReviewsV2 || isOutdatedReviewsV3 || isOutdatedAvailV3 || 
    isOutdatedJunePerfV2 || isMissingCritical || isVersionMismatch)
```

**风险**：代码不可维护，每次读这段代码都需要理解所有历史上下文

**建议**：只保留 `isVersionMismatch` 一条，删掉所有历史 `isOutdated*` 条件。因为版本号机制已经能覆盖所有历史更新场景——版本不匹配就重置，不需要逐条判断。

#### 3. Git 仓库 28MB（大量截图和大文件入库）

**现状**：以下大文件已提交到 Git 仓库：
- `daily-action-0617.png` — 3.3MB
- `data/full_attendance_raw.json` — 2.2MB
- `images/scenes/` 多张图片 — 各1.3-1.8MB
- `reports/weekly_review_0622.png` — 1.7MB
- `screenshots/` 20+张截图 — 每张~640KB

**风险**：clone 仓库慢，GitHub Actions 部署慢，协作困难

**建议**：
- 将 `screenshots/`, `reports/`, `output/`, `images/scenes/`, `data/*.json`（除代码需要的）加入 `.gitignore`
- 已入库的大文件可用 `git filter-repo` 清理历史（需要 force push）
- 截图和报告改为存本地，不进 Git

#### 4. 零 try-catch → 任何数据异常即白屏

**现状**：
- `app.js`：0 处 try-catch
- `pages.js`：仅 2 处 catch
- `Store.init()` 的 `JSON.parse(localStorage...)` 没有 try-catch 包装

**风险**：如果 localStorage 数据损坏（如用户手动修改、浏览器bug），页面直接白屏，无任何错误提示

**建议**：
```js
// Store.init() 加 try-catch
init() {
  try {
    const data = JSON.parse(localStorage.getItem(this.KEY));
    // ... 版本检查逻辑
  } catch (e) {
    console.error('数据解析失败，重置为默认值', e);
    localStorage.setItem(this.KEY, JSON.stringify(this.defaults));
  }
}
```
- 关键渲染函数（renderDashboard, renderRatings等）加 try-catch + 降级显示
- 全局 `window.onerror` 捕获，显示友好错误提示而非白屏

### P1 — 中优先级（下个迭代优化）

#### 5. 14个一次性脚本堆积

**现状**：`scripts/` 目录有 14 个废弃的一次性脚本（apply_june_v15.js, batch_update_v14.js, test_perf_v29.js 等），总共 300KB，无任何代码引用它们。

**建议**：移到 `scripts/archive/` 或直接删除。只保留 `fetch_linggong.js` 和可能复用的 `explore_linggong.js`。

#### 6. 195处内联style + 92行超200字符

**现状**：
- `pages.js` 中有 195 处 `style="font-size: ..."` 内联样式
- 92 行超过 200 字符（大量 HTML 模板拼接）

**风险**：难以统一修改样式、无法复用、代码可读性差

**建议**：
- 提取高频内联style为 CSS class（如 `font-size: 10px; color: var(--text-muted)` → `.text-caption`）
- 长 HTML 模板考虑用模板字符串或提取为函数

#### 7. Chart.js 实例不 destroy（内存泄漏）

**现状**：`pages.js` 创建了 Chart 实例（`new Chart`），但 0 处 `.destroy()`。用户在页面间切换时，旧 Chart 实例不会被回收。

**风险**：频繁切换页面 → Chart 实例堆积 → 内存占用增加 → 移动端可能卡顿

**建议**：在切换页面时调用 `chart.destroy()`，或改为单例模式复用。

#### 8. 无 loading 态 + 空状态仅3处

**现状**：
- 页面加载时无 loading/spinner/skeleton
- "暂无数据"类空状态提示仅 3 处（13个页面模块应该都有）

**风险**：数据量大时用户看到空白页面不知道是在加载还是出错了

**建议**：每个列表页加 loading skeleton + 空状态图。

### P2 — 低优先级（锦上添花）

#### 9. 安全性

- **前端认证无密码**：登录只需选姓名，sessionStorage 可手动伪造
- **1处 innerHTML**：有 XSS 风险（如果数据中有恶意脚本）
- **fetch_linggong.js** 中手机号通过命令行参数传入（相对安全，但 auth_state.json 不应入库）

**评估**：对于内部管理系统（13人使用），当前安全级别可以接受。如果未来开放给更多人使用，需要加密码认证。

#### 10. 操作流程繁琐

每次更新数据需要手动改3处：
1. `js/app.js` — 数据数组
2. `js/app.js` — `_dataVersion` + `DATA_VERSION`（两处都要改，且必须一致）
3. `js/pages.js` — 如果涉及评分逻辑

**建议**：
- 短期：写一个 update 脚本自动化版本号递增
- 中期：考虑数据与代码分离（JSON文件 + fetch加载）

#### 11. 数据管理：月度归档缺失

**现状**：所有月份数据混在一起。6月数据已完，7月数据要追加，但没有归档机制。

**建议**：月底运行归档脚本，将上月数据移到 `data/archive/2026-06.json`，app.js 中只保留当月+上月数据。

#### 12. fetch_linggong.js 跨月问题

**现状**：`--range=month` 在跨月时会返回当月数据而非上月，需要用 `--range=lastmonth` 或 `--start/--end`。

**建议**：脚本中加日期校验提醒——如果当前是月初1-3号，自动提示"是否获取上月数据？"。

## 三、做得好的地方

不是全是问题，也有很多亮点：

1. **评分逻辑已全面动态化** — 5维度评分（avail/perf/behavior/attendance/review）都从原始数据实时计算，不再硬编码分数，维护成本低
2. **版本号机制有效** — `_dataVersion` 双写 + `isVersionMismatch` 检查，确保用户总能拿到最新数据
3. **GitHub Actions 自动部署** — push 即上线，流程顺畅
4. **灵工打卡自动化** — fetch_linggong.js 实现了 API 拦截 + 自动登录 + 数据清洗，大幅减少手工录入
5. **响应式设计** — 支持移动端，虽然 @media 查询只有2处但布局自适应良好
6. **数据完整性** — 退换货追溯、员工缩写大小写不敏感匹配等细节处理到位
